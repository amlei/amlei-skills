# info-analyze

## 输入

1. **待分析文件**：已收集的信息文件路径，或直接提供的文本内容。
2. **分析目标/维度**：需要得出什么结论（如"核心能力要求是什么"、"匹配度如何"）。
3. **参照文件**（可选）：用于对比的基准文件路径。

## 交付

分析报告，写入 `.md` 文件。**若已有同名文件存在，先备份（带时间戳）再写入，不直接覆盖。** 返回：**文件路径** + **简短摘要**（所属领域、核心结论、信息缺口）。不返回报告正文。

结构：**分析目标 → 所属领域判定 → 各维度结论（判断 + 依据）→ 信息缺口 → 行动建议**。

## 规则

1. **拆解**：先通读输入内容，按其内在结构拆解为可独立审视的模块（如要求条目、能力维度、事实陈述、数据指标等）。不预设框架——内容长什么样就怎么拆。
2. **判断所属领域**：根据拆解结果判断内容所属领域/类型（如岗位 JD、竞品分析、技术方案、用户反馈等），明确标注判定理由。
3. **按领域分析**：用该领域的视角和标准分析拆解出的模块——不同领域关注的核心维度不同（JD 看能力模型与门槛，竞品看差异与定位，技术方案看可行性与风险）。分析深度匹配领域惯例，不套用无关框架。
4. **围绕目标提取**：只分析与目标相关的信息，不跑偏。每条结论都能追溯到来源。
5. **区分事实与推断**：明确写明的是事实；推导得出的是推断，须标注推理逻辑。
6. **指出信息缺口**：关键信息缺失时明确列出"还缺什么"。
7. **给出可行动建议**（按需）：结论指向下一步行动。

## 验收自检

交付前逐项确认，任何一项不通过则补齐后再交付：

- [ ] 已对输入内容做结构拆解，无遗漏关键模块
- [ ] 所属领域已判定并标注理由
- [ ] 分析维度与所属领域匹配，未套用无关框架
- [ ] 每条结论都有来源信息支撑，可追溯
- [ ] 事实与推断已区分标注
- [ ] 关键信息缺口已列出
- [ ] 结论直接回答了分析目标，未跑偏
- [ ] `.md` 文件已写入磁盘
- [ ] 返回内容只有文件路径 + 摘要，不含报告正文
