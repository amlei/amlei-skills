# 项目讨论（通用）

写简历和改简历都需要挖掘项目经历。本文件定义通用的项目讨论流程，写/改时共用。

## 项目类型

先帮用户区分项目类型，不同类型处理方式不同：

| 类型 | 说明 | 素材获取方式 |
|------|------|-------------|
| **公司项目** | 在职期间参与的工作项目 | 优先让用户给相关文档/材料；没有就跟用户聊 |
| **个人项目** | 业余自建、个人作品 | 优先看作品链接/作品集；没有就跟用户聊 |

素材来源不限形式：文档、PPT、报表、设计稿、代码仓库、运营数据、分析报告、作品链接等都可以。用户给到什么就读什么。

## 获取项目素材

**优先让用户提供项目相关材料**（网页链接、本地文件路径、文档、PPT、报表、设计稿、运营数据、分析报告等）。拿到后像探索一个系统一样深入梳理整条链路，不遗漏。

### 探索式梳理

目标不是"提取简历可用的部分"——而是**彻底理解用户做了什么**。像接手一个陌生系统一样，顺藤摸瓜，追问每一个疑点：

```
材料/文件 ──→ 读实际内容细节（非概览）──→ 提取关键信息
                │
                ├── 发现了什么？跟我预期一样吗？
                ├── 这个项目为什么存在？解决了什么问题？
                ├── 和哪些系统/团队相关？上下游是什么？
                ├── 有没有提到的链接、引用、依赖没看过？
                ├── 有哪些数字、指标、成果值得深挖？
                └── 哪里看不懂？需要问用户补全？
```

**操作方式（读到真实内容，不停在概览）：**

- 网页链接 → 链接若指向项目 / 文档 / 数据集等，**钻进它的实际内容**（仓库里的具体文件、文档全文、真实数据记录），不能只读 landing page / README / 摘要——那是概览，不是实际内容。追踪链接里点名的文件 / 章节，打开来读。
- 本地文件 → 读**真实内容**：文档类（.docx / .pdf / .pptx 等）用 markitdown 转全文；代码 / 配置 / 数据 / 设计稿等非文档文件，直接读源内容，不靠摘要或文件名猜测。

```bash
markitdown <文件路径> -o <输出路径.md>   # 仅用于文档类转换
```

- 目录 → 不止"浏览文件列表"——判断哪些关键后，**逐个读那些文件的实际内容**，不只看名字。

**原则：读真实细节、再取舍。** 概览 / README / 摘要 / 目录列表只是地图，不是地形——必须下钻到材料的实际内容，摸清背景、流程、关联、上下游的真实细节，把理解写进 `materials.md`，后续写简历再取舍。**停在概览 = 简历建立在不可靠的信息上，直接拉低准确性。**

### 带推理读材料 → 把线索转成待问问题

用户给的材料往往比他自述的更多——人会习惯性低报自己做过的事，尤其是不贴"主业标签"的那部分。所以别照单全收，要带推理去读：材料里凡是**暗示了用户没明说的工作、职责、经历**的细节，都是线索，转成待问问题记进 `materials.md` 的「待问疑点」区，留到[讨论补充](write-revise.md)环节问用户。不要替用户脑补，也不要漏掉。

可挖的线索维度（**只是示例，非穷举**——关键是带着"他还做过什么没说"的眼光去读，不要预设"某岗位就不该做别的活"）：

- **没明说的职责**：材料里出现用户没主动提的活（例：协作记录里别人的任务是你建的 → 你是不是还做了任务分配 / 协调？出现你经手的设计稿、文档、汇报 → 你还做了设计 / 文档 / 汇报？）
- **时间转折 / 空白**：不同时期领域、技术栈、岗位发生变化，或某段时间没记录 → 问怎么过渡、为什么转向（例：几个月前做 A 领域、后来转 B 领域，中间怎么转的）。这类转折常暴露成长决策和可迁移能力。
- **规模 / 协作范围**：经手的量级、对接的人数 / 角色，往往暗示比"执行者"更大的实际职责
- **任何没看明白、想进一步确认的细节**

如果用户表示没有材料可以给，那就跟用户一起聊这个项目，围绕以下几点挖掘：

1. **项目背景**：这个项目是做什么的？解决了什么问题？
2. **你的角色**：你负责了什么？是主导、核心参与还是协助？
3. **方法/过程**：用了什么方法或工具？为什么选它？
4. **难点与解决**：遇到最大的困难是什么？怎么解决的？
5. **量化成果**：带来了什么可量化的结果？（效率提升、用户增长、收入提升、覆盖率等）

## 素材结构

聊项目、查文件、访问项目地址过程中获取的内容，总结提炼后保存到对应简历目录，不丢失：

```
resume/{姓名}/{求职岗位}/
├── 简历.md
├── jd.md
├── materials.md          # 项目讨论 / 文件提炼 / 地址访问结果
└── files/                # 用户给的原始素材 + markitdown 转换结果都落盘到这里
```

- 用户给了文件（.docx/.pdf/.pptx/.xlsx）→ markitdown 转换，**转换后的 md 和原始文件都落盘到 `files/`**（不写临时目录），再摘要写入 `materials.md`
- 用户给了项目地址 → 访问核实 → 关键信息写入 `materials.md`
- 聊项目聊出的 STAR 素材 → 整理写入 `materials.md`

`materials.md` 可在下次改简历时作为素材源回溯，避免重复追问。

### 迭代更新

`materials.md` 不是写一次就定稿的。讨论过程中素材逐步清晰——用户补充了细节、纠正了说法、给出了量化数字——都应更新到 `materials.md` 对应条目，保持素材始终是最新、最准确的状态。
