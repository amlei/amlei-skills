# 写/改简历

从零写或迭代改写目标岗位简历。写和改本质相同——都是基于用户资料和岗位市场信息，对简历内容的改写操作。

## 收集就业市场信息

以 subagent 方式调用 amlei-job-market-research skill（输入：求职岗位 + 目标城市），产出市场报告（薪资、能力要求、行业趋势等），写入 `resume/{姓名}/{求职岗位}/市场报告.md`，作为后续简历定位与改写的依据。

## 分析用户项目资料

以 subagent 方式分析用户提供的项目资料（项目地址、项目文件），拆解用户做过什么、成果是什么。分析报告写入 `resume/{姓名}/{求职岗位}/分析.md`。

**⏸️ 人工确认**：展示分析报告，由用户确认准确性。用户指出遗漏或误读时，补充修正后再次确认。

**回写 profile（强制，最易遗漏）**：分析中挖到的真实能力 / 项目 / 成果，不能只写进 `分析.md` / `materials.md` 就交差——那两份是任务局部的。必须按 [profile.md](profile.md) 的写入流程，经评估 agent + 用户确认后回写到 profile（个人能力记忆），否则这些发现随任务结束而丢失，你下次会话又得从零认识用户。**项目资料探索（GitHub、博客、飞书知识库、文档等）是回写的高发场景**——分析完就触发回写，不要拖到「讨论补充」阶段甚至完全漏掉。

## 讨论补充（Human-in-loop）

分析确认后，进入持续讨论环节——围绕"用户在这份简历想呈现怎样的自己"。这一环节的核心不是等用户说，而是**主动追问**：用户记忆是碎片化的，聊到一半才想起来的事，你不问就永远沉没。

**三类必问：**

1. **把分析阶段的「待问疑点」逐条抛出**：[projects.md](projects.md) 带推理读材料时记下的疑点（角色越界、时间转折、规模暗示等），逐条直接问用户——"我看 XX 是你做的，你是不是还负责了 YY？"。这些疑点最易挖出隐性经历，不要放过。
2. **主动问"还做过什么"，从该领域的周边职能给角度**：别只笼统问"还有吗"（用户多半答不上）。结合目标岗位所在领域，把常被遗忘的周边职能逐个确认（**以下仅是举例，按实际领域替换**）：
   - 技术类周边：任务分配 / 排期、客户或需求对接、UI / 原型设计、跨部门协调、文档 / 带教
   - 其他领域类比：运营的跨部门 / 活动 / 数据分析、产品的竞品 / 需求走查、设计的规范 / 走查……
   - 关键是**给用户具体的钩子去回忆**，而不是开放空问。
3. **聊到新东西就即时落盘**：讨论中冒出的新能力 / 项目 / 成果（哪怕零碎），**当场写进 `materials.md`**——这是本次任务的发现记录，后续改写简历直接从这里取材；不写下来容易忘，挖到的就白挖了。值得跨岗位复用的，再评估 + 确认后写回 profile（个人能力记忆）。别攒到最后。

这三类追问**贯穿整个讨论，不是问一次就完**。每聊完一段经历，都再问一遍"这段之外还搭手做过什么吗"。

详见 [projects.md](projects.md)。

## 写作规则

**脱敏**：简历经历中涉及公司内部项目名称，一律用业务功能描述或规模定义替代（如"核心交易链路"、"支撑日均百万订单的订单系统"），严禁写入公司项目原文名称。不用"某"式占位表述。

**平实表述**：每句话让读者一眼看明白你具体做了什么。写每条经历时主动问用户"你当时具体是怎么做的"，引导他说出执行细节再落笔，不替用户编造。**不解释领域常识**：标准技术名词和概念（如"Skill""YAML frontmatter""Context Engineering"）是读者已知的领域常识，不在简历中解释它是什么。要么删掉无差异化的技术列举，要么只写你做了什么/做成了什么，不写"XX是什么"。

**链接限制**：仅 `self-intro` 模块可含外部链接（GitHub/作品集等），其他模块禁止出现链接。

**经历定义**：

- **self-intro**：头部模块——name / role / 联系方式 / avatar / links + education（教育背景作缩进子项），渲染成 Header。
- **个人简介**：独立 `# 个人简介` 模块，下接一段纯文本（渲染成 Summary）。不要把个人简介塞进 self-intro 当尾部文字——Header 里 trailing 段落会渲染异常。
- **工作/实习经历**（一家公司一个条目，保持"面"的完整）：
  - **基本盘**：总体职责描述 + 核心职责模块（3-5 条 bullet），含日常运维、团队协作、零碎事项汇总、公司内获奖等——回答"你日常负责什么"
  - **高光时刻**：在该条目下嵌入 1-2 个突出成果 bullet，带量化——回答"你在这家公司干成了什么"
  - 基本盘和高光时刻在同一条目内，不拆成独立项目
- **项目经历**（满足 4 条件：起止时间明确 + 可量化目标 + 核心角色 + 完整闭环结果）：
  - 分公司项目和个人项目，**公司项目在前，个人项目放后**，让工作经历和项目之间形成自然衔接感。

## 共创/迭代改写

默认逐节共创：按 个人简介 → 经历 → 技能 顺序，每节基于挖到的素材 + 市场报告出一版给用户确认 / 改，再下一节。要快就切「先整篇草稿再改」。

改已有简历时同理——看到已有 bullet，追问"这个项目你具体负责了什么"、"有没有遇到意外情况"、"结果能用数字说明吗"，帮用户把已有经历写得更具体、更量化。聊完一段经历后问"还有没有类似的项目"、"实习之前还做过什么相关的吗"，找出沉默亮点。

经历用 STAR + 量化，只放与岗位相关的；个人资料里的项目地址优先访问核实数据。

## 用户确认后再保存

每版内容经用户明确认可后，再写到 `resume/{姓名}/{求职岗位}/简历.md`。用户没说"可以/就这样"之前，绝不写入。
